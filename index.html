DevOps Upskilling — Starter Pack

A self-paced program. Work through modules in order. Each module has learning materials, an assignment, and acceptance criteria.

All materials are free and public.

Module 1

Linux

Terminal fluency and shell scripting.

Learning ~4–7hAssignment ~4–5hTotal ~8–12h

Learning materials

  • Essential: MIT Missing Semester
  • Essential: Linux Journey
  • Hands-on: OverTheWire Bandit (levels 0–20 min)
  • Additional: Linux Upskill Challenge
  • Reference: man7.org, tldr.sh

Assignment — recurring host check

Goal: recurring host-availability check that emails results.

Steps

  1. Shell script checks HTTP status of https://example.com and https://google.com. Emits OK/FAIL per host with timestamp.
  2. Email the result to your work inbox via any MTA / SMTP tool you choose. Credentials chmod-protected, never committed.
  3. Schedule via cron: single-line crontab entry, runs hourly, survives reboot.

Acceptance

  • Script idempotent
  • Runs after reboot (verified)
  • Emails arrive (screenshot or saved .eml)
  • No secrets in repo
Module 2

Git

Fundamentals, history rewriting, hooks, signing.

Learning ~3–5hAssignment ~3–5hTotal ~6–10h

Learning materials

  • Essential: Learn Git Branching (interactive)
  • Essential: Pro Git book
  • Additional: Conventional Commits, git worktree, GitHub Docs — signing commits with SSH

Assignment — rebase, hook, signing

Goal: practice history rewriting, hooks, and modern repo hygiene.

Steps

  1. Register a GitHub account with your 3PG email (skip if you have one).
  2. Create public repo devops-upskilling-<first>-<last>. No collaborators required.
  3. Push 5 dummy commits.
  4. Use git rebase --interactive: squash the first three, drop the fourth, reword the fifth.
  5. Push the rebased history without force-overwriting unrelated remote work — use --force-with-lease.
  6. Add a commit-msg hook that appends the keyword DevOps to commit messages.
  7. Configure SSH commit signing. Prove with git log --show-signature.
  8. Enable branch protection on main: require PR, require signed commits.

Acceptance

  • Rebased history visible on remote
  • Hook installed (show file)
  • --show-signature reports good signatures
  • Branch protection screenshot
Module 3

λ Programming — Functional Programming & Composition

Port your shell script to a real language, FP-first.

Learning ~4–7hAssignment ~8–13hTotal ~12–20h

Learning materials

  • Essential: Mostly Adequate Guide to Functional Programming
  • Essential: Grokking Simplicity (sample chapters)
  • Additional: refactoring.guru — Refactoring catalog, language-idiomatic style guide (PEP 8 / Effective Go / TS Handbook)
  • Reference: SOLID in Plain English — read but don't force OOP

Assignment — port the cron script, FP-first

Goal: reimplement the Module 1 logic in a real language with FP discipline.

Steps

  1. Port the Module 1 shell script to the language of your choice (Python / Go / TS / Elixir / …).
  2. FP discipline:
    • Pure functions for check logic (input → result, no I/O).
    • I/O (HTTP, email, time, env) at the edges only.
    • No shared mutable state.
    • Prefer map/filter/reduce (or language equivalent) over imperative loops where natural.
    • Immutable data structures.
  3. Compose checks: [Host] -> [CheckResult] via small composable functions.
  4. Linter + formatter wired (ruff / golangci-lint / eslint+prettier / …).
  5. Structured JSON logs (not print).
  6. Still scheduled by cron from Module 1.

Acceptance

  • lint clean
  • README explains the pure/impure boundary and the composition
Module 4

Continuous Integration (GitHub Actions)

Learning materials

  • Essential: Martin Fowler — Continuous Integration
  • Essential: GitHub Actions docs
  • Additional: GitHub OIDC to AWS, CodeQL docs, Trivy Action

Assignment — CI pipeline on the Module 3 repo

Goal: real CI on the Module 3 repo, not just one lint step.

Steps

  1. GitHub Actions workflow on PRs to main:
    • super-linter (or megalinter)
    • Language lint + tests with dependency cache (actions/setup-* + cache)
    • CodeQL SAST
    • Trivy repo scan (secrets, deps, IaC)

Acceptance

  • A failing PR (lint error) is blocked
  • OIDC role is assumable from the workflow
Module 5

Containers (Docker)

Learning ~4–6hAssignment ~6–10hTotal ~10–16h

Learning materials

  • Essential: Docker — Get Started
  • Essential: Dockerfile reference
  • Essential: Docker Compose v2 docs
  • Additional: Dockerfile best practices, Distroless images, Trivy image scanning
  • Reference: Julia Evans — Containers zine, Deep Dive into Docker Internals — Union FS

Assignment — containerize and compose

Goal: containerize the Module 3 app and run a two-container stack via Compose.

Steps

  1. Multi-stage Dockerfile for the Module 3 app:
    • Non-root USER.
    • HEALTHCHECK instruction.
    • Final image ≤ 100 MB (budget).
  2. compose.yaml (Compose v2) with two services:
    • app — runs the Module 3 check loop continuously (internal scheduler OR cron inside the container — document your choice).
    • sidecar — same image, monitors app, appends stats hourly to a named volume (not a host bind-mount).
  3. Image scanned with Trivy in CI (extend the Module 4 workflow). Fail on HIGH/CRITICAL.

Acceptance

  • docker compose up runs cleanly
  • docker image ls shows size ≤ 100 MB
  • Trivy gate visible in CI
  • Sidecar stats file populated in the volume
Module 6

Cloud — AWS + Terraform/OpenTofu

Biggest module. Re-read the AWS Free Tier cheat sheet first.

Learning ~8–12hAssignments ~12–18hTotal ~20–30h

Learning materials

  • Essential: AWS — What is Cloud Computing
  • Essential: AWS Skill Builder — Cloud Practitioner Essentials (free)
  • Essential: HashiCorp Learn — Terraform AWS
  • Essential: AWS Free Tier page
  • Additional: OpenTofu, AWS Well-Architected Labs, 5 Lessons from 300k lines of IaC

Assignment 1 — AWS account & static site

Goal: stand up a real public HTTPS site with no credit burn.

Steps

  1. Create an AWS free-tier account, enable MFA on root, create an IAM user for daily use.
  2. Write an index.html (hello world / bio).
  3. Private S3 bucket, with OAC (Origin Access Control) so only CloudFront can read.
  4. CloudFront distribution → S3 origin.
  5. HTTPS via the default *.cloudfront.net certificate (free). Bonus: own domain + ACM cert.
  6. Set a billing alarm at $5 via CloudWatch + SNS email.

Acceptance

  • Site loads over HTTPS
  • S3 not directly public
  • Billing alarm armed

AWS Free Tier: always-free at this scale (S3 5GB 12 mo, CloudFront 1TB always-free, ACM free).

Assignment 2 — IaC: pick a track

Pick one track. Document your choice and cost in the README. All tracks use Terraform/OpenTofu, remote state in S3 + DynamoDB lock table, and run fmt + tflint + checkov in CI.

Track A — Serverless always-free · recommended

  • Lambda (containerized — reuses Module 5 image) + API Gateway HTTP API + DynamoDB.
  • Deploy the Module 3 check logic as a Lambda triggered by an EventBridge schedule.
  • Cost: $0 ongoing at assignment scale.

Track B — Containers on Fargate Spot credit-burn

  • ECS cluster + ECS service on Fargate Spot + ALB.
  • Uses the Module 5 image.
  • Cost: ~$20–40 of the $200 credit if left running. Document teardown.

Track C — Containers on EC2 12-mo · classic

  • ECS cluster + ECS service on EC2 (t2.micro/t3.micro, single instance, 750 h/mo first 12 mo) + ALB.
  • Uses the Module 5 image.
  • Cost: ALB ~$16/mo is not free — eats ~$100 of the $200 credit over 6 months, unless you tear down daily. EC2 is free only the first 12 months.

Acceptance (all tracks)

  • terraform apply runs cleanly from scratch
  • Remote state in S3, lock works (try two concurrent applies)
  • CI runs fmt / tflint / checkov green
  • README documents track + cost + teardown command
Module 7

Continuous Delivery

Learning ~3–5hAssignment ~7–11hTotal ~10–16h

Learning materials

  • Essential: Dave Farley — Continuous Delivery YT
  • Essential: Martin Fowler — Continuous Delivery
  • Essential: semantic-release
  • Additional: GitHub Environments + manual approvals, Feature Toggles (Martin Fowler)

Assignment — CD pipeline

Goal: CD pipeline deploys the Module 6 app, separates release from deploy, and gates prod.

Steps

  1. GitHub Actions workflow on merge to main:
    • Build + push image (ECR for Tracks B/C) OR package Lambda (Track A).
    • terraform apply via OIDC role (no static keys, from Module 4).
    • Deploy to dev automatically.
  2. Separate release workflow on tag:
    • semantic-release reads conventional commits (Module 2).
    • Generates CHANGELOG.md.
    • Creates a GitHub Release with a version bump.
    • Release ≠ deploy.
  3. prod deploy: GitHub Environment, required reviewer (yourself), manual approval.
  4. Rollback strategy documented in the README.

Acceptance

  • Merge to main → dev deployed automatically
  • Tag push → release + CHANGELOG
  • Prod blocked until approval
  • Documented rollback
Final Project

🏆 Capstone — Status Page

Assemble Module 1–Module 7 into a real, deployed, self-monitoring status page.

Learning ~0h (reuses Module 1–Module 7)Assignment ~30–50hTotal ~30–50h

Architecture (Track A reference; B/C map analogously)

  • Frontend: SPA (any framework or vanilla JS), red/green dot per monitored host, last-check timestamp. Hosted on S3 + CloudFront (HTTPS, OAC, ACM cert).
  • Backend: Module 3 check logic.
    • Track A: Lambda + API Gateway HTTP API.
    • Track B: ECS Fargate Spot + ALB.
    • Track C: ECS on EC2 + ALB.
  • Cache: DynamoDB with TTL always-free — replaces ElastiCache Redis (not free).
  • Schedule: EventBridge cron triggers the check.
  • Observability: CloudWatch Logs · 1 custom metric (e.g. checks_failed_total) · 1 alarm → SNS email.
  • IaC: Terraform/OpenTofu, remote state S3 + DynamoDB, fmt + tflint + checkov in CI.
  • CI/CD: GitHub Actions, OIDC to AWS, multi-repo:
    • <name>-statuspage-frontend
    • <name>-statuspage-backend
    • <name>-statuspage-infra
  • Deploy: PR merge → dev auto; tag → semantic-release; prod manual approval (Module 7).

Acceptance

  • Public HTTPS URL serves the SPA
  • SPA reads from the backend API, shows red/green per host
  • Backend writes/reads cache from DynamoDB
  • CloudWatch alarm fires on simulated failure → email arrives
  • All 3 repos green CI, OIDC deploys, no static AWS keys
  • Teardown: terraform destroy cleans up all 3 stacks
  • Each repo's README documents track + cost estimate

Bonus

  • OpenTelemetry traces → CloudWatch (or Honeycomb free tier)
  • Status history page (last 24h from DynamoDB)
  • Multi-region health check
Final Checkout

🐞 Final Checkout · debugging — live faults on your deployed system

Learning ~3–5hAssignment ~7–15hTotal ~10–20h

How this module works

A program admin / instructor will introduce faults into your own deployed stack: app code, Terraform config, or AWS resource settings. Faults are not announced. You detect, diagnose, and fix.

Learning materials

  • Essential: Julia Evans — debugging blog & zines
  • Essential: Google SRE Book — Postmortem Culture
  • Essential: Brendan Gregg — Linux Performance Tools
  • Additional: Chrome DevTools docs, strace / ltrace man pages, tcpdump tutorial, CloudWatch Logs Insights query syntax, CloudTrail event history
  • Reference: The Debugging Mindset (ACM Queue)

Assignment — find & fix the planted faults

Fault categories (admin picks specifics)

  • Application bug — logic / concurrency / data-handling in backend code
  • Infrastructure bug — Terraform config drift / wrong IAM policy / wrong env var / wrong security group
  • AWS setup bug — manual console change outside IaC (e.g. modified Lambda alias, edited DynamoDB TTL, revoked role permission)

Steps per fault

  1. Detect — CloudWatch alarm, broken UI, failing API call, etc.
  2. Reproduce — minimal repro path documented.
  3. Diagnose — logs / metrics / CloudTrail / terraform plan drift / strace / browser devtools as appropriate.
  4. Fix via a PR with a regression test, or an alarm that would have caught it earlier.
  5. Blameless post-mortem markdown: timeline · impact · root cause · contributing factors · what went well · action items.

Acceptance

  • 3 PRs (one per fault), each linking repro + RCA + regression check
  • 3 post-mortem markdown docs, blameless language
  • A drift-detection alarm OR terraform plan CI check added to prevent the AWS-setup class of bug
